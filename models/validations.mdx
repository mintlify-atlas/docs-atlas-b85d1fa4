Sequelize has two distinct mechanisms for enforcing data integrity:

- **Validations** run in JavaScript, before any SQL is sent. Sequelize uses [validator.js](https://github.com/validatorjs/validator.js) as the underlying library.
- **Constraints** are enforced by the database engine itself (`NOT NULL`, `UNIQUE`, etc.).

<Note>
Validations only run when you call `Model.create()`, `instance.save()`, or `instance.validate()`. Bulk operations like `Model.bulkCreate()` skip per-row validations by default — pass `{ validate: true }` to enable them.
</Note>

## Validations vs constraints

| | Validations | Constraints |
|---|---|---|
| Where | JavaScript (Sequelize) | Database |
| When | Before SQL is sent | During SQL execution |
| Error type | `ValidationError` | Database error (may be wrapped) |
| Example | `isEmail`, `len`, custom functions | `NOT NULL`, `UNIQUE` |
| Can be skipped | Yes (`{ validate: false }`) | No |

Using both is recommended: validations give you clear error messages, constraints provide a database-level safety net.

## Built-in validators (validator.js)

Place validators in the `validate` key of an attribute's options. Each key is a validator name, and the value is either `true` or an array of arguments.

```typescript
{
  email: {
    type: DataTypes.STRING,
    validate: {
      isEmail: true,
    },
  },
  website: {
    type: DataTypes.STRING,
    validate: {
      isUrl: true,
    },
  },
  age: {
    type: DataTypes.INTEGER,
    validate: {
      isInt: true,
      min: 0,
      max: 150,
    },
  },
}
```

## Custom validators

A custom validator is a function that throws if validation fails. It receives the attribute's current value as its argument.

```typescript
{
  age: {
    type: DataTypes.INTEGER,
    validate: {
      isEven(value: number) {
        if (value % 2 !== 0) {
          throw new Error('Age must be an even number.');
        }
      },
    },
  },
}
```

To use a custom error message with a built-in validator, pass an object with a `msg` property:

```typescript
{
  email: {
    type: DataTypes.STRING,
    validate: {
      isEmail: {
        msg: 'Please provide a valid email address.',
      },
    },
  },
}
```

## Custom async validators

Return a `Promise` or accept a Node-style callback to write asynchronous validators.

```typescript
{
  username: {
    type: DataTypes.STRING,
    validate: {
      async isUniqueInExternalSystem(value: string) {
        const taken = await externalApi.isUsernameTaken(value);
        if (taken) {
          throw new Error('Username is already taken.');
        }
      },
    },
  },
}
```

## Model-level validators

Model-level validators run once per `save()` call and have access to all attributes via `this`. Define them in the `validate` option of `ModelOptions`.

```typescript
User.init(
  {
    startDate: { type: DataTypes.DATEONLY },
    endDate:   { type: DataTypes.DATEONLY },
  },
  {
    sequelize,
    validate: {
      datesInOrder() {
        if (this.startDate > this.endDate) {
          throw new Error('startDate must be before endDate.');
        }
      },
    },
  },
);
```

With the decorator approach, use the `@ModelValidator` decorator:

```typescript
import { ModelValidator } from '@sequelize/core/decorators-legacy';

class Event extends Model<InferAttributes<Event>, InferCreationAttributes<Event>> {
  declare startDate: string;
  declare endDate: string;

  @ModelValidator
  validateDates() {
    if (this.startDate > this.endDate) {
      throw new Error('startDate must be before endDate.');
    }
  }
}
```

## A model with multiple validators

```typescript
import {
  Model,
  InferAttributes,
  InferCreationAttributes,
  CreationOptional,
  DataTypes,
} from '@sequelize/core';

class User extends Model<
  InferAttributes<User>,
  InferCreationAttributes<User>
> {
  declare id: CreationOptional<number>;
  declare username: string;
  declare email: string;
  declare age: number;
  declare website: string | null;
}

User.init(
  {
    id: {
      type: DataTypes.INTEGER,
      primaryKey: true,
      autoIncrement: true,
    },
    username: {
      type: DataTypes.STRING(50),
      allowNull: false,       // DB constraint
      unique: true,           // DB constraint
      validate: {
        notEmpty: true,       // Sequelize validation
        len: [3, 50],
        isAlphanumeric: true,
      },
    },
    email: {
      type: DataTypes.STRING,
      allowNull: false,
      validate: {
        isEmail: {
          msg: 'Must be a valid email address.',
        },
        notEmpty: true,
      },
    },
    age: {
      type: DataTypes.INTEGER,
      allowNull: false,
      validate: {
        isInt: true,
        min: 13,
        max: 120,
      },
    },
    website: {
      type: DataTypes.STRING,
      allowNull: true,
      validate: {
        isUrl: true,
      },
    },
  },
  {
    sequelize,
    validate: {
      // model-wide: no website if age < 18
      ageGate() {
        if (this.age < 18 && this.website != null) {
          throw new Error('Users under 18 cannot set a website.');
        }
      },
    },
  },
);
```

## ValidationError

When one or more validations fail, Sequelize throws a `ValidationError` containing an array of `ValidationErrorItem` objects.

```typescript
import { ValidationError, ValidationErrorItem } from '@sequelize/core';

try {
  await User.create({ username: 'x', email: 'not-an-email', age: 10 });
} catch (err) {
  if (err instanceof ValidationError) {
    for (const item of err.errors) {
      console.log(item.path);      // attribute name
      console.log(item.message);   // error message
      console.log(item.type);      // 'Validation error', 'notNull Violation', etc.
      console.log(item.value);     // the invalid value
    }
  }
}
```

## Database-level constraints

Constraints are declared in the attribute options and result in SQL `DDL` clauses being added to the `CREATE TABLE` statement (or enforced by the database independently of Sequelize).

### allowNull: false

Adds a `NOT NULL` constraint to the column and also runs a pre-SQL null check.

```typescript
{
  email: {
    type: DataTypes.STRING,
    allowNull: false,    // -> NOT NULL in SQL
  }
}
```

### unique: true

Adds a `UNIQUE` constraint. Pass a string to create a named composite unique index:

```typescript
// Single column unique
{ email: { type: DataTypes.STRING, unique: true } }

// Composite unique on (firstName, lastName)
{
  firstName: { type: DataTypes.STRING, unique: 'fullname_unique' },
  lastName:  { type: DataTypes.STRING, unique: 'fullname_unique' },
}
```

### primaryKey and autoIncrement

```typescript
{
  id: {
    type: DataTypes.INTEGER,
    primaryKey: true,
    autoIncrement: true,
  }
}
```

<Tip>
Combine both validations and constraints for the best developer experience: validations produce friendly error messages before any SQL is sent, while constraints protect the database in case validations are bypassed.
</Tip>
